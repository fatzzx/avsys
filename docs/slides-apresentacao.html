Introdução

Definição do Problema

⚠ Double-Booking
Dois usuários obtêm reservas válidas para o mesmo assento — anomalia crítica de corretude que impacta diretamente a experiência do passageiro
↔ Lost Update
Sem coordenação, atualizações de múltiplos processos se sobrescrevem silenciosamente — a segunda escrita invalida a primeira
🔀 Race Condition
Dois processos leem "available" → passam na validação → ambos escrevem com sucesso — resultado incorreto sem erro explícito

Introdução

Motivação

Sistemas monolíticos resolvem o problema com transações globais e locks pessimistas. Microsserviços impõem novos desafios:

  • Bancos de dados independentes por serviço — não existe transação distribuída global
  • Múltiplas réplicas do mesmo serviço processam requisições sem coordenação implícita
  • Teorema CAP (Brewer) — num sistema distribuído não é possível garantir ao mesmo tempo consistência, disponibilidade e tolerância a partições de rede

Teorema CAP (Brewer, 2000)

C
Consistência
A
Disponibilidade
P
Tolerância a partições

Num sistema distribuído só é possível garantir dois dos três

💡 Nossa decisão de projeto
Garantimos consistência + tolerância a partições, abrindo mão de disponibilidade total — em falha de rede, o sistema rejeita a reserva (409) em vez de arriscar confirmar o mesmo assento duas vezes

Introdução

Objetivos do Projeto

  1. Arquitetura de microsserviços com fronteiras de dados claras, cada serviço proprietário exclusivo do seu banco e comunicação assíncrona via eventos
  2. Prevenção de double-booking via lock distribuído Redis + versionamento otimista SQLite — defesa em duas camadas independentes
  3. Comunicação assíncrona com RabbitMQ para desacoplar o processamento de pagamentos do fluxo de reserva, reduzindo latência percebida
  4. Containerização completa com Docker Compose — health checks, orquestração de dependências e volumes persistentes
3 Microsserviços Redis Lock RabbitMQ Docker Compose SQLite WAL Bun · ElysiaJS · TypeScript

Introdução

Relevância da Solução

🏭 Padrões de Produção
Redis distributed lock e optimistic locking são usados em escala por Amazon, Booking.com e Airbnb — a solução replica práticas estabelecidas da indústria
🛡 Defense in Depth
Duas barreiras independentes: mutex Redis elimina contenção antes de acessar o banco; versionamento SQLite captura casos extremos como expiração de TTL
⚡ Latência < 120 ms
Pipeline completo de reserva (lock + transação + evento) em ~120 ms — o pagamento é assíncrono e não bloqueia a resposta ao cliente
📐 Generalização
Os padrões aplicados são válidos para qualquer sistema de reserva com alta concorrência — hotéis, shows, consultas médicas, vagas de estacionamento

  • Padrão Microsserviços — cada serviço encapsula um subdomínio de negócio, tem banco próprio e é independentemente implantável
  • NGINX como API Gateway — roteamento por prefixo, rate limiting (20 req/s/IP), balanceamento least_conn

  • REST síncrono + AMQP assíncrono — síncrono onde o cliente aguarda, assíncrono para efeitos colaterais entre serviços

Metodologia

Tecnologias Utilizadas

⚡ Bun + ElysiaJS
Runtime JS/TS nativo — executa TypeScript sem transpilação. 2,5–3× mais rápido que Node.js (TechEmpower 2024). ElysiaJS: validação TypeBox end-to-end com tipos gerados automaticamente.
🗄 SQLite (bun:sqlite)
Driver nativo embutido, zero dependências. WAL mode: leituras concorrentes durante escritas. PRAGMA foreign_keys e busy_timeout. Versionamento otimista com coluna version.
🔴 Redis (ioredis)
Lock distribuído: SET NX EX — atômico em single-thread. Cache: buscas TTL 5 min. Idempotência: pagamentos TTL 24 h. Release via script Lua.
🐰 RabbitMQ (amqplib)
Topic exchange airline.events. Filas duráveis, prefetch=1, Dead Letter Exchange para mensagens com falha. Routing key no formato {agregado}.{evento}.

Organização dos Microsserviços

✈ flight-catalog
:3001

Inventário de voos, assentos, preços e horários.

Cache Redis 5 min por parâmetro de busca

Consome: reservation.confirmed, reservation.cancelled

DB: airlines, airports, flights, seats

🔒 reservation-service
:3002

Orquestra o fluxo crítico de reserva com lock e transação atômica.

Redis lock + SQLite versionamento otimista

Publica: payment.requested, reservation.confirmed/cancelled

DB: reservations, seats (cópia local), events

💳 payment-service
:3003

Consumidor assíncrono — processa pagamentos sem bloquear a reserva.

Idempotência via Redis (TTL 24 h) — sem duplicatas em re-entrega

Taxa de sucesso: 90% (simulado)

DB: payment_orders, notifications

Estrutura dos Contêineres e Banco de Dados

6 Contêineres Docker Compose

nginx flight-catalog reservation-service payment-service redis rabbitmq
  • depends_on + service_healthy — NGINX só sobe após todos os serviços passarem no health check
  • Volumes nomeados para persistência dos três bancos SQLite entre reinicializações
  • 2 redes isoladas — Redis e RabbitMQ invisíveis ao gateway
flight-catalog → flights.db reservation-service → reservations.db payment-service → payments.db Configurações SQLite (todos os serviços): journal_mode = WAL ← leituras concorrentes foreign_keys = ON ← integridade referencial busy_timeout = 5000 ← retry automático em lock Controle de concorrência: coluna version + UPDATE … WHERE version = ? → changes = 0 ⟹ ROLLBACK (anomalia detectada)
